## 使用规则

- 完成度按“退出标准”计算，不按投入时长或代码行数计算。
- 每周日更新一次当前完成度、证据、指标和下一周目标。

- Commit 要能说明真实增量，不为保持连续记录提交无意义改动。

## 下一阶段计划

### 07-20 ～ 07-26：M2 检索评测与质量改进

目标：先证明搜索质量，再决定加什么 Tool。

- 建立 15～20 条脱敏或公开仓库评测问题，至少覆盖：符号、错误信息、配置键、路径过滤和语言过滤；
- 每条样例记录预期 `repo`、`path` 和可接受命中范围；
- 输出 `Hit@1`、`Hit@5`、端到端闭环成功率和搜索延迟；
- 对重复命中、无关结果、上下文过长分别记录失败案例；
- 只根据失败数据选择一项改进，例如去重、排序、查询构造或上下文预算；
- 将评测变成一条可重复执行的命令，并保留改进前后的结果。

注意：Zoekt 是文本/正则搜索。没有 AST、LSP 或语言解析证据时，不把候选命中宣称为准确的“定义”或“引用”。如果要增加工具，优先命名为 `search_symbol_candidates`，或在返回结果中明确这是候选结果。

### 07-27 ～ 08-16：M3 Code Understanding Agent

目标：从“有 MCP 工具”升级为“有可评测的 Agent”。

- 先在现有仓库的 `examples/` 中实现最小 Agent Client，稳定后再决定是否拆仓；
- 使用模型 Tool Calling，让模型自主选择 `search_code` 和 `get_file_context`；
- 设置最大 Tool 调用次数、超时、错误处理和完整调用轨迹；
- 输出结构化答案：`answer`、`evidence`、`uncertainties`、`next_queries`；
- 每条证据必须包含 `repo/path:line`，没有证据时明确回答信息不足；
- 用至少 10 个未参与 Prompt 调试的问题验证 Agent；
- 暂不引入 LangGraph。只有简单 Tool Loop 已无法清晰表达状态和恢复逻辑时，再记录理由后引入。

M3 退出指标：

- 检索 `Hit@5 >= 80%`，若未达到则先修检索，不用 Prompt 掩盖；
- 引用中的文件和行号有效率 `>= 95%`；
- 每次任务不超过 6 次 Tool 调用；
- 失败时能说明是无命中、服务不可用、上下文不足还是模型判断失败。

### 08-17 ～ 09-06：M4 Incident Copilot

- 准备至少 5 个可重复案例：空指针、配置缺失、超时、依赖调用失败、发布后回归；
- 对比“只问模型”和“检索真实代码后回答”的结果差异；
- 生成一段不超过 3 分钟的完整演示。
