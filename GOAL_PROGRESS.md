## 使用规则

- 完成度按“退出标准”计算，不按投入时长或代码行数计算。
- 每周日更新一次当前完成度、证据、指标和下一周目标。
- 同一时间只推进一个主里程碑；当前里程碑未达到退出标准前，不开新框架或新项目。
- Commit 要能说明真实增量，不为保持连续记录提交无意义改动。

## 目标岗位验收标准

岗位名称不是判断依据。满足以下条件的 Offer 才计为北极星目标完成：

- 工作内容以代码、系统设计、工程化和线上结果为主，而不是主要编写 Prompt、配置低代码 Workflow 或重复交付业务场景；
- 团队有真实用户、生产流量或可追踪的业务/产品指标，而不是只以“上线了多少场景”衡量进展；
- 岗位实际覆盖以下能力中的至少 3 项：
  - Agent Loop、Tool/MCP 接入与执行；
  - Session、状态持久化、Checkpoint 与失败恢复；
  - Trace、Eval、实验和线上失败诊断；
  - Sandbox、工具权限、审计或 Human-in-the-loop；
  - Context Engineering、Memory 或长任务上下文管理；
  - 成功率、质量、Token、成本和延迟优化；
- 团队对 Agent Runtime/Harness 或关键工程层有实际所有权，不完全依赖外部低代码平台；
- 金融岗位可以作为第一次转型入口，但必须沉淀可迁移的 Agent 系统能力；通用岗位不要求一定是 Code Agent 产品；
- 薪资、城市、通勤、团队稳定性等个人底线在 M6 前量化，未满足个人底线的 Offer 不计为“满意 Offer”。

注意：Zoekt 是文本/正则搜索。没有 AST、LSP 或语言解析证据时，不把候选命中宣称为准确的“定义”或“引用”。如果要增加工具，优先命名为 `search_symbol_candidates`，或在返回结果中明确这是候选结果。

### 07-27 ～ 08-16：M3 Code Understanding Agent + 最小 Harness

目标：从“有 MCP 工具”升级为“有可评测、可解释的最小 Agent Harness”。Harness 在本项目中指模型周围负责上下文构造、Agent Loop、Tool 路由、状态事件和 Trace 的控制层，不另造通用框架。

- 先在现有仓库的 `examples/` 中实现最小 Agent Client，稳定后再决定是否拆仓；
- 定义最小的 `Session`、`Step`、`ToolCall`、`ToolResult` 和 `FinalAnswer` 事件结构，为每次运行分配稳定的 `task_id`；
- 实现 Context Builder，明确系统指令、当前任务、可用 Tool Schema、已有证据和剩余预算如何进入模型上下文；
- 使用模型 Tool Calling，让模型自主选择 `search_code` 和 `get_file_context`；
- 通过统一 Tool Router 完成参数校验、调用、结果标准化和错误分类，不让模型直接依赖具体 MCP Client 实现；
- 设置最大 Tool 调用次数、超时和明确终止条件，完整记录模型、Tool 和状态转换事件；
- 输出结构化答案：`answer`、`evidence`、`uncertainties`、`next_queries`；
- 每条证据必须包含 `repo/path:line`，没有证据时明确回答信息不足；
- 用至少 10 个未参与 Prompt 调试的问题验证 Agent；
- 暂不引入 LangGraph。只有简单 Tool Loop 已无法清晰表达状态和恢复逻辑时，再记录理由后引入。

M3 退出指标：

- 检索 `Hit@5 >= 80%`，若未达到则先修检索，不用 Prompt 掩盖；
- 引用中的文件和行号有效率 `>= 95%`；
- 每次任务不超过 6 次 Tool 调用；
- 每次运行均可按 `task_id` 查看按顺序排列的模型与 Tool 事件、耗时和终止原因；
- 失败时能说明是检索无命中、Tool/服务不可用、Harness 约束、上下文不足还是模型判断失败。

### 08-17 ～ 09-06：M4 可恢复 Incident Copilot

目标：把线上工程经验变成项目差异点，并用 Incident Copilot 的真实失败驱动 Harness 的持久化、恢复和安全边界设计。

- 输入脱敏日志、异常栈和故障描述；
- 用 Structured Output 提取异常类、方法、错误文本、服务名和可能相关配置；
- 调用 Code Search MCP 定位候选代码并读取上下文；
- 将任务状态和事件流持久化；进程退出或 Tool 临时失败后，能够从最后一个已完成步骤恢复，而不是重新执行整个任务；
- 增加 Tool 超时、有限重试和失败注入，至少覆盖服务不可用、超时和返回异常三类故障；
- 为 Tool 标注 `read`、`write`、`execute` 风险级别。当前只读检索可自动执行；只有出现真实写入或诊断执行需求时才增加审批，不为演示 Human-in-the-loop 人造高风险 Tool；
- 输出“事实证据、原因假设、置信度、待补信息、排查步骤”，不把假设写成事实；
- 准备至少 5 个可重复案例：空指针、配置缺失、超时、依赖调用失败、发布后回归；
- 对比“只问模型”和“检索真实代码后回答”的结果差异；
- 生成一段不超过 3 分钟的完整演示。

M4 退出指标：

- 至少保存 1 次任务执行中断、重启进程并成功恢复的可复核证据；
- 三类失败注入均能在 Trace 中得到正确分类，并按策略重试、停止或请求人工处理；

- Trace 能区分模型判断失败、Tool 失败、Harness 失败和评测器问题。

## 每周时间预算

结合 10:30～21:00 的工作时间，默认采用约 10～12 小时的可持续节奏：

| 时间 | 安排 |
| --- | --- |
| 周一、周二、周四 | 60～75 分钟主功能开发 |
| 周三 | 45～60 分钟测试、文档或修复 |
| 周五 | 休息；最多做 20 分钟轻量复盘 |
| 周六 | 4～5 小时完成本周主要可演示增量 |
| 周日 | 3～4 小时测试、README、日报复盘和下周拆解 |

每周最低交付不是“学习了多少”，而是：

- [ ] 一个可以演示的增量
- [ ] 4～6 个有意义的 Commit
- [ ] 测试保持全绿
- [ ] 一条设计决策或失败记录
- [ ] 一次指标更新
- [ ] 周日更新本进度表

## 项目约束

- 不把公司代码、内部日志、域名、仓库名、API Key 或本地私有路径放进公开材料；
- `.secrets/` 继续保持 Git 忽略，并在公开前检查提交历史；
- 不为了“像 Agent”而增加 Multi-Agent、Memory、RAG 或复杂工作流；
- 不把 Harness 当作新的框架名称或第二个项目；只在现有 Agent 中实现能对应真实失败、可通过指标验证的控制机制；
- Agent Harness、MCP Tool 和 Eval Harness 分层设计：Tool 提供能力，Agent Harness 负责执行与控制，Eval Harness 负责隔离运行、记录和评分；
- 没有不可信代码执行或高风险写操作时，不为了展示技术名词强行增加 Sandbox 或审批流程；
- 新增依赖和框架必须能对应一个已经出现的真实问题；
- 作品集以一个旗舰项目为主，只有面试反馈证明存在明确缺口时才增加第二个完整项目。
